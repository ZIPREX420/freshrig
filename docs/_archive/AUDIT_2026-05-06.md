### 1.2 Headline architectural assessments

1. **The platform abstraction is the right shape but not load-bearing.** The Windows / Linux / macOS command trees are legitimately different (WMI vs `/proc` vs `system_profiler`), but a lot of cross-cutting glue (process spawning, error mapping, registry-vs-file-vs-defaults reads) is duplicated three times. A traits-based `SystemBackend` with three impls would compress ~1200 LOC and make the macOS gap (currently all stubs) less daunting to fill.
2. **The frontend is in noticeably better shape than the backend.** Strict TS, centralized design tokens, FOUC-prevention done correctly, hotkeys wired via `react-hotkeys-hook`, and a sane error boundary at the root. The biggest frontend issues are bundle bloat (recharts/canvas-confetti/zxcvbn loaded eagerly) and store granularity.
3. **The hardening story is half-built.** Crash-log scrubbing, panic hooks, CRLF-aware `.gitattributes`, `audit.toml` suppressions with documented reasons, weekly Dependabot, SBOM generation — all good. But CSP, code signing, subprocess timeouts, capability tightening, headless-mode authentication — all missing. This is the gap between "engineering hygiene" and "production hardening."
4. **CI is decent but Windows-biased.** Linux is in the matrix and runs, but the SBOM step is Windows-only, and there's no `cargo audit`, no test reporting, no coverage gate, no artifact upload of the actual Linux installer for smoke-testing.
5. **The macOS code path is dead weight in this state.** All commands return `Err("macOS support coming soon")`, CI doesn't compile macOS, and the frontend doesn't gate on `usePlatform().isMacos` consistently. Either commit to shipping macOS (re-enable CI, fill stubs) or cfg-gate the macOS routes out of the frontend so users see "not supported" instead of bare error toasts.

### 1.3 Highest-impact improvements

If you pick one week of focused work, do these in this order:

1. Set CSP, sign the NSIS installer, add `cargo audit` to CI. (R1, R3, R8)
2. Add a `with_timeout` subprocess helper, retro-apply to all `Command::new` call sites. (R5)
3. Slice `appStore`; centralize `invokeWithToast`. (R10, R11)
4. Document and write tests for the LemonSqueezy variant-id allowlists so flipping pre-launch → live is one-line. (R2)
5. Re-sync `CLAUDE.md` + memory to the actual v2.2.0 state.

Everything else listed below is real, but those five are the leverage points.

### 2.2 Architecture style

Modular monolith. Single Tauri process; webview frontend talks to Rust backend via Tauri IPC (`@tauri-apps/api`'s `invoke`). All persistent state in:

- `tauri-plugin-store` JSON (`settings.json`)
- `rusqlite` DB (machine inventory for fleet)
- `keyring` (license, integration secrets)
- Filesystem (profiles, snapshots, logs)

No backend service, no network protocol, no microservice split. The architecture is correct for the product (offline-first desktop tool); no need to introduce services.

### 2.4 Entry points and execution flow

- **GUI start:** `main.rs::main()` → if no CLI flags, call `lib.rs::run()` → Tauri `Builder` registers plugins, tray, panic hook, command handler → window mounts → `index.html`'s FOUC script applies accent → React mounts (`main.tsx`) → `App.tsx` renders → effects fire (load settings, schedule update check, license revalidate, drift toast).
- **Headless apply-profile:** `freshrig.exe --headless --task=apply-profile --profile-id=<id>` → `lib.rs::run_headless_apply_profile` writes `pending-apply.json` → exits → next GUI launch picks up the marker.
- **Headless contracts / SMART check:** Two more headless entry points exposed via `pub use` in `lib.rs:11-12` for fleet automation.

### 2.5 Command surface (IPC)

## 3. Full findings

Findings are grouped by phase and severity. Each entry: ID, severity, title, where, root cause, fix, effort, risk.

### 3.1 Architecture (Phase 2)

#### A-1 — `Result<T, String>` everywhere is a context-loss anti-pattern  ⏤ MED
- **Where:** all of `src-tauri/src/commands/`
- **Root cause:** Every command returns `Result<T, String>`. Errors lose type, lose chain, lose source. When `get_hardware_summary` fails on a slow WMI machine, the frontend receives `"WMI query timed out"` but cannot tell the user *which* WMI query (CPU? GPU? Net?), or whether it's recoverable.
- **Fix:** Define a single `enum FreshRigError` with variants (`Wmi`, `Registry`, `Subprocess`, `Network`, `License`, `Io`, `Validation`), implement `Display`, derive `Serialize` so it crosses the IPC boundary as structured JSON. Use `thiserror` for the boilerplate. Add `From<X>` impls so `?` propagates cleanly.
- **Effort:** ~6 h. Mostly mechanical.
- **Risk:** Low — additive, no behaviour change for happy paths. Frontend can keep treating errors as strings during migration.

#### A-2 — Three parallel command trees with no shared trait  ⏤ MED
- **Where:** `commands/`, `commands/linux/`, `commands/macos/`
- **Root cause:** `cleanup`, `hardware`, `network`, `services`, `apps`, `report` exist three times. Each pair shares ~50–70% structural skeleton (signature, DTO, error path). Today the only shared abstraction is in `commands/linux/util.rs` (used by Linux only) and `commands/macos/util.rs` (used by macOS only). Windows has no equivalent — its utility helpers (registry reads, WMI helpers) are scattered inside command modules.
- **Fix:** Introduce a `SystemBackend` trait in `platform/`:
  ```rust
  pub trait SystemBackend {
      async fn list_services(&self) -> Result<Vec<Service>, FreshRigError>;
      async fn set_service_start_type(&self, name: &str, t: StartType) -> Result<(), FreshRigError>;
      async fn run_cleanup(&self, plan: &CleanupPlan) -> Result<CleanupReport, FreshRigError>;
      // ...
  }
  ```
  Implement `WindowsBackend`, `LinuxBackend`, `MacosBackend`. Tauri commands become thin: `commands::services::get_services` calls `state.backend.list_services().await`. Total LOC drops; macOS gap becomes "implement one impl" rather than "fill 11 stub files."
- **Effort:** 3–5 days incremental (migrate one feature per PR).
- **Risk:** Medium — requires careful handling of WMI's blocking nature (`spawn_blocking`).

#### A-3 — `commands/mod.rs` registry must stay manually in sync with `lib.rs`  ⏤ LOW
- **Where:** `src-tauri/src/commands/mod.rs`, `src-tauri/src/lib.rs:132+`
- **Root cause:** `tauri::generate_handler!` is a macro array of function paths. New command? Add to both `mod.rs` and the handler list. No compiler enforcement.
- **Fix:** Move command registration into a `register_handlers!` declarative macro per module that auto-collects:
  ```rust
  // commands/services.rs
  freshrig_command!(get_services, set_service_start_type, ...);
  ```
  Or simpler: a unit test that compares `cargo expand` output against a manifest.
- **Effort:** ~3 h.
- **Risk:** Low.

#### A-4 — Frontend ↔ backend type drift is a lurking risk  ⏤ MED
- **Where:** `src-tauri/src/models/*.rs` ↔ `src/types/*.ts`
- **Root cause:** Rust DTOs use `#[serde(rename_all = "camelCase")]`; TS interfaces are hand-written. There's no enforcement that adding a field on one side changes the other. `serde(default)` on every field forgivingly accepts missing fields, which is *good* for forward-compat but hides schema drift.
- **Fix:** Generate TS types from Rust using `ts-rs` (annotation crate). One `cargo test --features ts-export` writes `.ts` to `src/types/__generated__/`. Add CI check that tree is clean after generation.
- **Effort:** ~1 day.
- **Risk:** Low.

### 3.2 Security (Phase 5)

#### S-1 — `csp: null` in `tauri.conf.json`  ⏤ CRIT
- **Where:** `src-tauri/tauri.conf.json:25`
- **Root cause:** Tauri docs explicitly warn that `null` disables CSP. With CSP off and capabilities granting `shell:allow-execute` + `shell:allow-spawn`, a single XSS in any rendered string would pivot directly into running arbitrary commands as Administrator.
- **Fix:** Set a strict CSP. Tauri v2 supports `default-src 'self'; img-src 'self' data: asset: https://asset.localhost; ...`. Iteratively enable until the app loads. The webview only needs to talk to itself (no third-party APIs from the frontend — all IPC goes through Rust), so the CSP can be tight. Sample baseline:
  ```json
  "csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' asset: data:; connect-src 'self' ipc: http://ipc.localhost; font-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'"
  ```
  Note: `style-src 'unsafe-inline'` is needed because the FOUC script in `index.html` and Tailwind v4 inject inline styles. If you can eliminate that (use a class on `<html>` instead of inline `style.setProperty`), you can drop `'unsafe-inline'` for styles too.
- **Effort:** ~2 h to set, ~2 h to validate every page.
- **Risk:** Medium — wrong CSP bricks the UI. Iterate via `tauri dev` console warnings.

#### S-2 — License validation accepts any FR-XXXXX-XXXXX shape  ⏤ CRIT (intentional pre-launch)
- **Where:** `src-tauri/src/commands/license.rs:20-29`
- **Root cause:** `EXPECTED_STORE_ID = 0`, `EXPECTED_PRODUCT_ID = 0`, both variant arrays empty. The comment block at lines 9–19 documents the pre-launch posture honestly. The validate path (further down the file) treats `0` as "skip check" so dev keys work.
- **Fix:** Pre-launch checklist in code:
  1. Add a `debug_assertions || cfg(feature = "dev-license")` gate so non-zero IDs are required in release builds. Compiler enforces it.
  2. Add a unit test `assert_pro_variants_nonempty_in_release()`.
  3. Bake the public-key check for the LemonSqueezy webhook into validate (so a network MITM can't forge a "valid" response). Currently the code trusts whatever `https://api.lemonsqueezy.com/v1/licenses/validate` returns over rustls; adding signature verification on the JSON body is defense-in-depth.
- **Effort:** ~3 h once you have the real IDs.
- **Risk:** None — flipping placeholders is safe; the test gate makes "ship without setting them" a CI failure.

#### S-3 — Capabilities grant broad shell scope to the webview  ⏤ HIGH
- **Where:** `src-tauri/capabilities/default.json:9-11`
- **Root cause:**
  ```json
  "shell:allow-execute",
  "shell:allow-spawn",
  "shell:allow-stdin-write",
  ```
  These let the *webview JavaScript* invoke `window.__TAURI__.shell.execute(...)` directly. Combined with `requireAdministrator` and CSP=null, any XSS becomes a SYSTEM-level RCE.
- **Fix:** Audit whether the frontend actually uses `shell.execute` or `shell.spawn` from JS. Grep for `tauri-apps/plugin-shell` imports in `src/`. If all subprocess work routes through `invoke('command_name')` (Rust does the spawning) — which is the right architecture — then drop these capabilities entirely. Frontend can keep `tauri-plugin-opener` (for opening URLs/files) without `tauri-plugin-shell`'s execute.
- **Effort:** ~2 h to audit + remove.
- **Risk:** Low if no frontend code imports `@tauri-apps/plugin-shell`. The plugin is in `package.json:41`, so verify usage.

#### S-4 — Headless `pending-apply.json` has no integrity / ACL  ⏤ HIGH
- **Where:** `src-tauri/src/lib.rs:21-39`
- **Root cause:** `run_headless_apply_profile` writes `{ profileId, queuedAt }` to `pending-apply.json` with default file permissions, then the next GUI launch reads and applies it. Local privilege escalation: a non-admin user writes a malicious marker; admin launches FreshRig, which honours it.
- **Fix:** (1) HMAC-sign the payload with a key from the keyring. (2) Set ACL/permissions on the marker so only the launching user can write it (Windows: `icacls` after `std::fs::write`; Unix: `0600`). (3) On read, verify HMAC; if bad, log + delete + ignore.
- **Effort:** ~4 h.
- **Risk:** Low.

#### S-5 — Subprocess shell-injection surfaces  ⏤ HIGH
- **Where:** `src-tauri/src/commands/services.rs` (apply_start_type), `network.rs` (set_dns_servers), various PowerShell-via-`format!` callers
- **Root cause:** `format!("Set-DnsClientServerAddress -ServerAddresses (@({}))", dns_ips.join(","))` — any caller passing an IP that contains `'); ...; (` corrupts the script. Some sites (`services.rs`) defensively `.replace('\'', "''")`; others don't. There is no central "build a PowerShell argv" helper.
- **Fix:** Either:
  - **(a)** Use Tauri's tokio Command API with explicit args (`-Command`, `Set-...`, args as separate `args`) — PowerShell parses each arg without re-shell-interpolating.
  - **(b)** Use the WMI/CIM API directly (`MSFT_DNSClientServerAddress`) and skip PowerShell entirely.
  - Add validation (`is_valid_ipv4_or_ipv6`) at the command boundary regardless.
- **Effort:** ~1 day to retro-apply.
- **Risk:** Low — caller validation is purely additive.

#### S-6 — Machine fingerprint is admin-spoofable  ⏤ MED
- **Where:** `src-tauri/src/commands/license.rs` (fingerprint helper)
- **Root cause:** `SHA256(MachineGuid + CPU ID + SMBIOS UUID)`. `MachineGuid` lives in `HKLM\SOFTWARE\Microsoft\Cryptography\MachineGuid` and any admin can rewrite it, invalidating the fingerprint and allowing license-key reuse on the same physical machine.
- **Fix:** This is a known tradeoff in licensing. Mitigations:
  - Add SMBIOS BIOS-UUID as a separate hash component (already done — keep it).
  - Add MAC of the primary NIC (changeable, but not by admin alone).
  - Bind to a hardware-rooted secret (TPM EK certificate hash). Heavyweight.
- **Effort:** ~6 h for MAC inclusion; ~3 days for TPM.
- **Risk:** Low. Most users are honest; this is mitigation, not prevention.

#### S-7 — `secrets.rs` keyring access doesn't authenticate the requesting context  ⏤ LOW
- **Where:** `src-tauri/src/commands/secrets.rs`
- **Root cause:** Any code in the FreshRig process can read any keyring entry. If an attacker can inject a Rust crate (supply-chain) or compromise the Tauri webview (CSP=null + shell:allow-execute), they get all secrets.
- **Fix:** Compounds with S-1, S-3. Once those are tight, this is acceptable.
- **Effort:** Bundled with S-1/S-3.

#### S-8 — `tauri.conf.json` updater single endpoint, no fallback  ⏤ MED
- **Where:** `src-tauri/tauri.conf.json:73-78`
- **Root cause:**
  ```json
  "endpoints": ["https://github.com/.../latest/download/latest.json"]
  ```
  GitHub Releases is the single point of failure. If GitHub blocks in user's region (China, certain enterprise networks) or has an outage, no updates land. Pubkey is correctly embedded.
- **Fix:** Add at least one mirror — Cloudflare R2 or a small static origin. Tauri's updater will fall through endpoints in order.
- **Effort:** ~2 h once a mirror exists.
- **Risk:** None.

#### S-9 — NSIS installer is unsigned  ⏤ HIGH
- **Where:** `.github/workflows/release.yml`
- **Root cause:** No Authenticode signing step. End users see SmartScreen "Windows protected your PC" warning; conversion drops sharply, and trust collateral builds for "FreshRig is malware" reviews.
- **Fix:** Add a `signtool` step in `release.yml` after `tauri-action`:
  ```yaml
  - name: Sign installer
    if: matrix.platform == 'windows-latest'
    run: |
      & "C:/Program Files (x86)/Windows Kits/10/bin/x64/signtool.exe" sign /f $env:CERT_FILE /p $env:CERT_PASS /tr http://timestamp.digicert.com /td sha256 /fd sha256 src-tauri/target/release/bundle/nsis/*.exe
    env:
      CERT_FILE: ${{ runner.temp }}/cert.pfx
      CERT_PASS: ${{ secrets.WINDOWS_CERT_PASS }}
  ```

### 3.4 Performance (Phase 4)

#### P-1 — Subprocess calls have no timeout  ⏤ HIGH
- **Where:** `src-tauri/src/util.rs::silent_cmd`, `src-tauri/src/commands/linux/util.rs::run_cmd`, every `Command::new` site
- **Root cause:** `cmd.output()` blocks until the child exits. WMI on a sick disk; smartctl on a wedged USB drive; winget waiting on slow internet — all hang indefinitely. The frontend command sits in `pending` and the user thinks the app froze.
- **Fix:** New helper:
  ```rust
  pub async fn run_with_timeout(
      mut cmd: Command,
      timeout: Duration,
  ) -> Result<Output, FreshRigError> {
      let mut child = cmd.spawn()?;
      tokio::select! {
          out = child.wait_with_output() => out.map_err(Into::into),
          _ = tokio::time::sleep(timeout) => {
              let _ = child.kill().await;
              Err(FreshRigError::Timeout)
          }
      }
  }
  ```
  Default timeout 30s; per-call override (`winget install` needs 5+ min). Frontend gets `Timeout` variant, can offer "Retry."
- **Effort:** ~1 day to write helper + retro-apply to ~40 call sites.
- **Risk:** Medium — careless timeouts kill long-running winget installs. Pick generous defaults; let installers configure.

#### P-2 — Sequential winget install loop blocks GUI  ⏤ MED
- **Where:** `src-tauri/src/commands/apps.rs::install_apps` (and the Linux/macOS twins)
- **Root cause:** Apps installed one-by-one. 10 apps × 30s = 5 minutes of "is it working?" UX.
- **Fix:** Two options:
  - **(a)** Stream progress events. Each install emits `app::install::progress` via `app_handle.emit_all`. Frontend already has `InstallProgressPanel`; wire it.
  - **(b)** Parallel installs (3-4 concurrent) using `tokio::stream::FuturesUnordered`. Risk: winget locks the package store; concurrent installs may fail.
  - Combine: serial winget calls but emit per-app progress.
- **Effort:** ~6 h.
- **Risk:** Medium for parallel; low for streaming.

#### P-3 — `recharts` re-renders on every parent state change  ⏤ LOW
- **Where:** `src/components/dashboard/SmartHealthCard.tsx` (and similar)
- **Root cause:** No `memo`/`useMemo` on chart `data` arrays. Re-renders even when `data` reference is unchanged.
- **Fix:** `useMemo(() => transform(rawData), [rawData])`; `React.memo` the chart component.
- **Effort:** ~1 h per chart.
- **Risk:** None.

#### P-4 — `jwalk`/`trash` for cleanup may iterate millions of files synchronously  ⏤ MED
- **Where:** `src-tauri/src/commands/cleanup.rs`, `commands/linux/cleanup.rs`, `commands/macos/cleanup.rs`
- **Root cause:** Walking `%TEMP%`, browser caches, `~/.cache/` can be hundreds of thousands of files. `jwalk` is parallel walking, good — but the *enumeration* result is one big `Vec<PathBuf>` returned to frontend, which then re-renders. UI freezes.
- **Fix:** Stream entries via Tauri events. Backend emits `cleanup::scan::progress` chunks of 1000 paths at a time. Frontend updates incrementally.
- **Effort:** ~6 h.
- **Risk:** Low.

### 5.1 Subprocess timeout helper (P-1 fix)

**Before** (`src-tauri/src/util.rs`):
```rust
#[cfg(target_os = "windows")]
pub fn silent_cmd(program: &str, args: &[&str]) -> Result<String, String> {
    use std::os::windows::process::CommandExt;
    let output = std::process::Command::new(program)
        .args(args)
        .creation_flags(0x08000000) // CREATE_NO_WINDOW
        .output()
        .map_err(|e| format!("spawn {}: {}", program, e))?;
    if !output.status.success() {
        return Err(String::from_utf8_lossy(&output.stderr).to_string());
    }
    Ok(String::from_utf8_lossy(&output.stdout).to_string())
}
```

**After** (`src-tauri/src/runtime/timeout.rs`):
```rust
use std::time::Duration;
use tokio::process::Command;
use crate::error::FreshRigError;

pub async fn run_with_timeout(
    program: &str,
    args: &[&str],
    timeout: Duration,
) -> Result<String, FreshRigError> {
    let mut cmd = Command::new(program);
    cmd.args(args);

    #[cfg(target_os = "windows")]
    {
        use std::os::windows::process::CommandExt;
        cmd.creation_flags(0x0800_0000);
    }

    let mut child = cmd
        .stdout(std::process::Stdio::piped())
        .stderr(std::process::Stdio::piped())
        .spawn()
        .map_err(|e| FreshRigError::Subprocess {
            program: program.to_string(),
            source: e.to_string(),
        })?;

    let result = tokio::time::timeout(timeout, child.wait_with_output()).await;

    match result {
        Ok(Ok(output)) if output.status.success() => {
            Ok(String::from_utf8_lossy(&output.stdout).to_string())
        }
        Ok(Ok(output)) => Err(FreshRigError::SubprocessExit {
            program: program.to_string(),
            code: output.status.code(),
            stderr: String::from_utf8_lossy(&output.stderr).to_string(),
        }),
        Ok(Err(e)) => Err(FreshRigError::Subprocess {
            program: program.to_string(),
            source: e.to_string(),
        }),
        Err(_) => Err(FreshRigError::Timeout {
            program: program.to_string(),
            after: timeout,
        }),
    }
}
```

Old `silent_cmd` callers migrate one-at-a-time; default timeout 30s; long-running commands (winget install) override.

import { invoke } from "@tauri-apps/api/core";
import { toast } from "sonner";

export async function invokeOrToast<T>(
  command: string,
  args?: Record<string, unknown>,
  options?: { silent?: boolean; rethrow?: boolean }
): Promise<T | null> {
  try {
    return await invoke<T>(command, args);
  } catch (raw) {
    const msg =
      typeof raw === "string"
        ? raw
        : raw && typeof raw === "object" && "message" in raw
          ? String((raw as { message: unknown }).message)
          : String(raw);
    if (!options?.silent) {
      toast.error(`${command} failed`, { description: msg });
    }
    if (options?.rethrow) throw raw;
    return null;
  }
}

### 6.4 Release script

`scripts/release.ts` (or `release.ps1`) — automate the 4-file version bump, regenerate Cargo.lock, commit with conventional message, tag, push. Eliminates the "did I bump all four?" risk.

```ts
#!/usr/bin/env tsx
// scripts/release.ts
import { execSync } from "node:child_process";
import { readFileSync, writeFileSync } from "node:fs";

const bump = process.argv[2] as "patch" | "minor" | "major";
if (!["patch", "minor", "major"].includes(bump)) {
  console.error("Usage: tsx scripts/release.ts <patch|minor|major>");
  process.exit(1);
}

const pkg = JSON.parse(readFileSync("package.json", "utf8"));
const [maj, min, pat] = pkg.version.split(".").map(Number);
const next =
  bump === "major" ? `${maj + 1}.0.0`
  : bump === "minor" ? `${maj}.${min + 1}.0`
  : `${maj}.${min}.${pat + 1}`;

console.log(`Bumping ${pkg.version} → ${next}`);

// 1. package.json
pkg.version = next;
writeFileSync("package.json", JSON.stringify(pkg, null, 2) + "\n");

// 2. src-tauri/Cargo.toml
let cargo = readFileSync("src-tauri/Cargo.toml", "utf8");
cargo = cargo.replace(/^version = ".*"$/m, `version = "${next}"`);
writeFileSync("src-tauri/Cargo.toml", cargo);

// 3. src-tauri/tauri.conf.json
const conf = JSON.parse(readFileSync("src-tauri/tauri.conf.json", "utf8"));
conf.version = next;
writeFileSync("src-tauri/tauri.conf.json", JSON.stringify(conf, null, 2) + "\n");

// 4. src/config/app.ts
let app = readFileSync("src/config/app.ts", "utf8");
app = app.replace(/APP_VERSION = "[\d.]+"/, `APP_VERSION = "${next}"`);
writeFileSync("src/config/app.ts", app);

// 5. Cargo.lock
execSync("cargo generate-lockfile --manifest-path src-tauri/Cargo.toml", { stdio: "inherit" });

console.log(`Done. Now: git add -A && git commit -m "chore: release v${next}" && git tag v${next} && git push --follow-tags`);
```

## 7. Alternative approaches (decision table)

### 7.1 Cross-platform process abstraction

| Approach | Pros | Cons | Verdict |
|---|---|---|---|
| **Status quo** (parallel command trees, no shared trait) | Easy to reason about per-platform; OS-specific code is right next to its usage | High duplication; macOS gap is 11 stub files; refactor cost is sunk into every new feature | Move on |
| **Shared trait (`SystemBackend`)** | Single feature impl across 3 platforms; macOS gap shrinks to "implement one impl"; clear seam for testing with mock backend | Async traits in Rust are ergonomic enough now (1.75+) but still ceremony; trait drives toward single-implementation locking | **Recommended** |
| **Macros for command dispatch** | Lowest LOC | Compile-time complexity; harder to debug | Not recommended |
| **Plugin architecture** (each feature is a Tauri plugin) | Future-proof for community plugins | Overkill for solo project; Tauri's plugin story is more for cross-app reuse than intra-app modularity | Not now |

### 7.2 Error type

| Approach | Pros | Cons | Verdict |
|---|---|---|---|
| **`Result<T, String>`** (status quo) | Simple; serializes for free | Loses context; can't pattern-match in frontend | Move on |
| **`thiserror::Error` enum** | Clean variants; `?`-propagates; serialize with serde | One more crate; some migration cost | **Recommended** |
| **`anyhow::Error`** | Quickest path | Doesn't serialize; lose variant info on IPC boundary | Use only inside backend, not on command return |

### 7.3 Type generation

| Approach | Pros | Cons | Verdict |
|---|---|---|---|
| **Hand-written TS** (status quo) | Zero tooling; clear ownership | Drift risk; no compile-time guarantee | Move on |
| **`ts-rs`** | Single source of truth; CI-enforced | Annotation churn on every DTO | **Recommended** |
| **OpenAPI spec + codegen** | Industry standard | Heavyweight for a desktop IPC layer | Not recommended |
| **`specta` (Tauri-native)** | Designed for Tauri | Less mature than ts-rs as of last check | Evaluate |

### 7.4 State management

| Approach | Pros | Cons | Verdict |
|---|---|---|---|
| **Zustand sliced stores** (after Q-1 fix) | Already adopted; cheap; selector-friendly | Manual selector discipline | **Recommended** |
| **Zustand slices pattern** (one store, slice methods) | Single store, scoped slices | Still re-renders on neighbor slice changes | Not as good |
| **Jotai/Recoil** | Atomic granularity | Migration cost | Not now |
| **TanStack Query for server state** | Caching, refetch, dedup of `invoke` calls | Adds dep; useful only if you expand network usage | Worth it after license/update flows grow |

### 7.5 Testing

| Approach | Pros | Cons | Verdict |
|---|---|---|---|
| **vitest + RTL + Playwright + tauri-driver** | Modern; fast; e2e covers IPC | Setup cost ~3d | **Recommended** |
| **Jest** | Established | Slow with ESM/Vite; vitest is a drop-in better | Not recommended |
